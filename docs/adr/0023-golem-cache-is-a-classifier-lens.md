# GOLEM's cache is a classifier-lens, transparent to both value and time

GOLEM//Console v3 renders Poxim's unit-3 cache (Harvard, 8 lines × 2 sets, 4-word blocks,
LRU-by-AGE). Unlike a textbook cache, this one **never sits between the CPU and memory**: the
value a load returns comes from memory as it always did, and the cache only *classifies* each
access as a Hit or a Miss and counts the ratio. It is a lens, not a data path — and this is not a
GOLEM simplification, it is what the reference emulator actually does, which we replicate rather
than "fix".

## Why not a real cache

Because the reference `poxim3` is the oracle, and its cache is a classifier by construction — the
hit/miss trace and statistics are generated by a structure that runs *beside* an always-truthful
memory, not one that serves data. Two consequences of that fall out, and both are deliberate:

- **Transparent to value.** The cached `data` words exist only to *detect* a hit — the reference's
  hit test is `tag matches AND cached-word == memory-word`. A load's result is read from memory,
  never from the cache. Modelling the cache as a data path would be modelling a machine that does
  not exist.
- **Transparent to time.** A Miss costs **no cycles**. The reference's main loop advances the
  Watchdog and FPU one tick per instruction whether an access hits or misses; the cache never
  touches that loop. Adding a miss penalty — however pedagogically tempting, since "a miss is slow"
  is the whole point of caches — would slide the Watchdog countdown and the FPU dispatch, and
  `3_watchdog_cache.out`, `3_interruption_cache.out` and `3_fpu_cache.out` would stop diffing clean.
  The Step stays the machine's only unit of time (ADR 0018); the *cost* of a miss lives in the
  presentation layer (a visual beat, the Clock breathing), never in `step`.

## Consequences

- **The reference's write quirks are replicated, not corrected.** Write-through, no write-allocate,
  and — the load-bearing one — a write hit does **not** update the cached word, so the value goes
  stale against memory and the next read of that address is classified as a Miss. This is woven into
  the hit/miss pattern the oracle records; a "helpful" fix would go red across every program that
  stores. Each deviation-from-textbook is written down in `ISA.md`, same statute as the broken `ble`
  (ADR 0019) — a red diff must always mean a decision, never a mystery.
- **The cache is not a Device.** It is not memory-mapped, the program cannot address or arm it, and
  it never interrupts. The glossary marks it as its own category to keep it from being modelled like
  the Watchdog or FPU (`CONTEXT.md`).
- **The cache is a mode, default on, fixed at Machine creation.** `cache on`/`cache off` are
  presentation commands like `clock`, allowed only while no Machine exists — so a trace is never
  half-wrapped and the two oracles (`.out` without the lens, `_cache.out` with it) each stay
  diffable. With the mode off, `step` behaves exactly as v2: no cache field, no cache events.
- **This closes the Poxim trilogy.** Units 1–3 are the whole of the project that ships a reference
  oracle. Later course topics (virtual memory, pipeline, superscalar) are lectures with no emulator,
  no fixtures, no oracle — and GOLEM's statute is oracle-driven (ADR 0019). The product's ceiling is
  therefore explicit: **no oracle, no feature.**
